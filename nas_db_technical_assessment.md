# 工程師技術分析：使用實體 NAS 作為網頁應用程式資料庫的可行性與架構評估

本文件從系統架構師與網路安全工程師的角度，深入評估將生產環境（Production）網頁應用程式的資料庫（如 PostgreSQL / MySQL）部署於公司實體 NAS，並與外部雲端主機（如 Vercel）對接之可行性。

---

## 一、 技術評估核心摘要

| 指標 | 評估狀態 | 技術說明 |
| :--- | :--- | :--- |
| **技術可行性** | **可實現 (Feasible)** | 可透過 Port Forwarding、DDNS / 固定 IP，將 NAS 中的資料庫連接埠暴露至公網以供雲端主機連線。 |
| **架構建議度** | **強烈不推薦 (Anti-Pattern)** | 嚴重違反現代雲端安全防護與無伺服器（Serverless）架構設計原則。 |
| **核心瓶頸** | **網路延遲 (Latency)**、**安全性 (Security)**、**可用性 (Availability)** | 連線路徑過長，且直接暴露資料庫核心端口至公網具備極高安全威脅。 |

---

## 二、 核心技術瓶頸分析

### 1. 網路延遲與連線池瓶頸 (Network Latency & Connection Pooling)
* **冷啟動與 RTT 延遲**：我們的網頁端託管於雲端（如 Vercel Serverless Functions），其節點位於全球 CDN（例如 AWS 香港 `ap-east-1` 或美東 `us-east-1`）。每一次資料庫查詢（Query）都必須跨越公網連回公司內部 NAS，去回程（RTT）延遲通常高達 **50ms ~ 150ms**，而同機房雲端資料庫僅需 **1ms ~ 5ms**。這會造成網頁加載速度極慢。
* **Serverless 連線耗盡**：Serverless 運算為無狀態（Stateless），每次觸發都會建立新連線。若 NAS 連線速度慢，連線無法快速釋放，會導致 TCP 連線數迅速達到資料庫上限（Connection Limit），造成 `Too many connections` 崩潰。

### 2. 資安漏洞與攻擊面擴大 (Security & Attack Surface)
* **暴露核心端口**：使用 NAS 當資料庫，必須在公司閘道器（Gateway）設定虛擬伺服器，將 `5432` (PostgreSQL) 或 `3306` (MySQL) 對公網開放。這會引來網路上黑客機器人的暴力破解與掃描。
* **勒索軟體連帶風險**：NAS 通常為公司內部檔案伺服器（SMB / NFS 共享）。若資料庫伺服器遭 SQL Injection 或零日漏洞（Zero-day）攻破，駭客將取得 NAS 系統權限，進而對整個辦公室內網進行橫向滲透（Lateral Movement），導致公司機密檔案被加密勒索。
* **防護方案成本高**：若要安全連接，必須建立 IP 增強白名單、自建 WireGuard / Tailscale VPN 隧道，或透過 Cloudflare Tunnels 進行反向代理。這會大幅增加網路配置複雜度。

### 3. 可用性與容災機制 (High Availability & Disaster Recovery)
* **單點故障 (SPOF)**：辦公室的停電、斷網（ISP 機線維護）、人為誤觸 NAS 電源，都會導致資料庫當機。
* **備份與損耗**：資料庫有高密度的 I/O 讀寫需求。一般 NAS 所用之機械硬碟（HDD）或非企業級固態硬碟（SSD），在頻繁的 web 併發讀寫下，故障率會顯著上升。

---

## 三、 建議架構：雲端生產環境 + 本地 NAS 備份（折衷方案）

若主管基於「資料產權」或「備份需求」希望將資料留在 NAS 中，建議採用**混合式架構（Hybrid Architecture）**：

```
[ 前台網頁 (Vercel) ]
       │ (1ms 極速查詢)
       ▼
[ 雲端託盤資料庫 (Supabase / Postgres) ] ◄──┐
       │ (每日排程同步)                      │ (安全隔離)
       └─────────────────────────────────────┘
       ▼
[ 公司實體 NAS (唯讀 / 僅做冷備份備存) ]
```

### 操作步驟：
1. **線上環境**：網站一樣使用雲端資料庫（如 Supabase）進行高併發、低延遲的讀寫。
2. **自動備份至 NAS**：在公司內部 NAS 設定一條排程腳本（Cron Job），每天凌晨自動執行 `pg_dump`，透過 API 安全地將雲端資料庫的備份檔（`.sql`）拉回本機 NAS 封存。
3. **優勢**：既能確保網站 24 小時快速、安全運作，又能滿足主管「實體資料儲存在公司硬碟」的安全感。
